# Android NFCタグ認識に向けた研究開発準備計画

## 1. 目標定義
成果物は、以下を実現できる実運用可能なシステムとして定義する。
- Android端末でNFCタグを検出できる
- タグペイロードを解析・検証できる
- 安全な業務アクションを実行できる
- 追跡可能な運用ログを記録できる

## 2. スコープ基準
対象範囲（In-scope）:
- Androidアプリ（NFCスキャンとユーザーフロー）
- APIバックエンド（タグ検証とアクション認可）
- データモデル（タグ台帳、操作履歴、担当者記録）
- セキュリティ制御（署名、リプレイ防止、監査ログ）
- 実環境に近い条件での検証

フェーズ1での対象外（Out-of-scope）:
- 既存の全レガシーシステムとの完全統合
- 要求がない限り高度な耐クローン専用ハードウェア
- iOSネイティブ対応（別フェーズで計画可能）

本フェーズ前提:
- プラットフォームはAndroidを主対象とする。
- iOSは本フェーズでは対象外（NG）として扱う。

## 3. ステークホルダーヒアリング項目
実装前に以下の回答を収集する。
- タグスキャン後に開始される業務は何か
- オフライン運用は必要か
- スキャンから処理完了までの許容遅延上限は何か
- 公式サポート対象の端末とOSバージョンは何か
- 求めるセキュリティ水準は何か（UID照合のみか、署名付きペイロードか）
- 法務・コンプライアンス監査に必要なログ要件は何か

## 4. 技術判断ゲート
ゲートA: タグ戦略
- UIDのみ戦略（高速だがセキュリティは弱い）
- NDEFペイロード戦略（中程度）
- 署名付きNDEFペイロード戦略（本番推奨）

ゲートB: アプリ動作モード
- Foreground Dispatchのみ（簡易）
- 最適化ポーリング付きReader Mode（推奨）

ゲートC: バックエンド依存度
- 常時オンライン必須
- ローカル署名トークン検証を併用するオフライン優先

ゲートD: データガバナンス
- ログ保持期間
- 個人情報マスキング方針

## 5. PoC成功基準
以下を満たした場合にPoC成功と判定する。
- テスト条件下でタグ読み取り成功率が99%以上
- スキャンから処理完了までの中央値が1.5秒以下
- リプレイ攻撃が遮断される
- 改ざんペイロードが拒否される
- 誰がいつ何をしたかをログで再構成できる

## 6. 推奨準備シーケンス
1. 正確なユースケースと脅威前提を定義する。
2. NFCタグ種別とペイロード形式を選定する。
3. Androidスキャナープロトタイプを構築する。
4. バックエンド検証APIを実装する。
5. 通常条件と悪条件の両方で現地試験を実施する。
6. 本番移行計画を作成する。

## 6.1 検証媒体と優先順位
優先順位:
1. 購入済みNFCタグ（読取・書込PoCの主対象）
2. IC認証可能なカード式名刺（読取可否と仕様依存範囲の確認）
3. マイナンバーカード（法令順守前提で検出/接続安定性中心の検証）

判定方針:
- 媒体ごとに「読取可」「書込可」「制約あり」を明示して記録する。
- 個人情報・機微情報を含む可能性がある媒体はログ最小化を徹底する。

## 7. 主なリスクと対策
リスク: タグ複製またはペイロード改ざん
- 対策: 暗号署名 + サーバー側nonce/リプレイ制御

リスク: 端末断片化による読み取り挙動の不安定化
- 対策: 認定端末リスト + 端末モデル別ベンチマークスクリプト

リスク: 運用現場のネットワーク不安定
- 対策: キャッシュ + リトライキュー + フォールバック手順

リスク: 人的オペレーションミス
- 対策: シンプルなUX、明示的確認、教育SOP
