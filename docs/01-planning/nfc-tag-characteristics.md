# NFCタグ特性まとめ（研究共有用）

## 1. 目的
この文書は、研究メンバー間でNFCタグの仕様上の制約と運用上の判断ポイントを共有するためのメモである。

## 2. 対象タグの前提（NTAG213）
- 13.56MHz帯のNFCタグ
- ユーザーデータ領域は一般に144バイト
- NDEF形式でスマートフォンから読み書き可能
- 量産タグでは、書き込み保護やロック状態が個体ごとに異なる場合がある

注記:
- 販売情報で「書き込み可能」とされていても、実運用では個体状態（ロック済み、保護設定）で書き込めないケースがある。

## 3. 書き込み可否の基本
- 事前に100%確定するのは難しく、実際に書き込みを試行して判定するのが基本
- 判定は「書き込み成功レスポンス」だけでなく「直後の再読み取り一致」で確定する
- 失敗時は以下で分類する
  - 権限拒否
  - タグ非対応
  - 読み取り専用（ロック済み）
  - 近接不良/タイミング不良

## 4. ロック機能の考え方
### 4.1 ロックの種類
- 永久ロック（不可逆）
  - ロック後は基本的に書き換え不可
- パスワード保護（タグ機能依存）
  - 書き込み・読み取りに制限をかける

### 4.2 重要な注意点
- 永久ロックは元に戻せない前提で扱う
- 本番タグをロックする前に、必ずテストタグで手順を確定する
- ロック前チェックリストを運用に組み込む

## 5. Web NFC（ブラウザ）でできること/できないこと
### できること
- NDEFの読み取り
- NDEFテキスト書き込み
- 直後の再読み取りによる照合

### できないこと（または制約が強いこと）
- 低レベルAPDU制御
- セキュア領域への直接アクセス
- タグの詳細ロック設定操作（実装や環境に依存）

## 6. 書き込み不可だった場合の代替運用
- UID（シリアル番号）をキーにサーバーDB照合する運用が現実的
- ただし、UID単独運用は複製耐性が弱いため、重要用途では追加対策が必要

推奨対策:
- UID + サーバー側状態管理（有効/無効、最終利用時刻、レート制限）
- 異常アクセス監視（短時間の多発、連続失敗）
- 高セキュリティ用途は署名付きデータや上位タグへ移行

## 7. 研究時の標準検証手順（簡易）
1. タグを1枚のみ手元に置く（混線防止）
2. 一意文字列を書き込む（例: test-YYYYMMDD-HHMMSS）
3. 同一端末で直後に再読み取りする
4. UIDと内容一致を記録する
5. 必要時のみロック操作を実施し、前後比較を保存する

## 9. URL起動時に送れる情報の整理（重要）
結論:
- NFCタグをかざしてURLを起動しただけでは、端末がUID（シリアル番号）を自動でURLへ付与して送ることは基本できない。

理由:
- OSはタグ内のNDEF URLを開くだけであり、読み取り時の低レベル情報をURLパラメータへ自動注入しない。

実務上の対応:
- タグURLに識別子トークンを埋め込む（例: https://example.com/nfc?t=RANDOM_TOKEN）。
- URL先のサーバーでトークン照合し、登録情報（任意コード等）を取得する。
- 高い真正性が必要な場合は、専用アプリで再読取して追加情報を送信する。

## 10. Android/iOS共通化の推奨方式
推奨方式:
- タグにはHTTPS URLのみを格納する。
- Webサービス側で端末判定、権限判定、業務分岐を行う。
- 必要時のみネイティブアプリへディープリンクする。

メリット:
- Android/iOSで共通仕様を維持しやすい。
- 将来アプリ追加時もURL基盤を流用できる。
- タグ運用（発行、失効、監査）をサーバー主導で管理できる。

最低要件:
- 推測困難なトークンを使用する。
- トークン失効管理（有効/無効）を実装する。
- アクセス監査ログを保存する。
